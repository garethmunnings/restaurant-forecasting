# Spur Restaurant Revenue Forecasting

Forecast **July 2025 daily sales** for 22 Spur restaurants and present the results in a manager-facing dashboard.

Built for the Stratech Signal Intelligence Student Programme. The scenario assumes **today is 30 June 2025** — July has not happened yet, so the forecast is a true out-of-sample projection.

---

## What this project produces

1. **`july_forecast.csv`** — the deliverable forecast: exactly **682 rows** (22 restaurants × 31 days), with columns `restaurant_id`, `date`, `predicted_revenue` (ZAR, ≥ 0). Generated by the forecasting pipeline at [forecast/output/july_forecast.csv](forecast/output/july_forecast.csv).
2. **Manager dashboard** — a local React/Vite app showing historical sales and the July forecast, switchable per restaurant, with financial KPIs aimed at non-technical restaurant managers.

## Data

All source files live in `data/`, which is **git-ignored** because the raw sales file exceeds GitHub's 100 MB limit. You must supply these files locally before running anything. Revenue is in **ZAR**.

| File | Purpose |
|------|---------|
| `spur_reporting.student_program.sale.csv` (~1.7M rows) | One row per bill. `amount` is the bill total. **The only source for revenue sums.** History ends 30 Jun 2025. |
| `spur_reporting.student_program.restaurant.csv` | 22 sites: `restaurant_id`, `brand` (all SPUR), `province`. |
| `spur_reporting.student_program.sale_customer.csv` | Bill-to-customer links. For count/signal features only — **never sum revenue from this file** (double-counts). |
| `spur_reporting.student_program.customer.csv` | Optional customer profile features. |
| `sa_school_holidays.csv` | SA school + public holidays 2023–2025, with inland/coastal clusters. |
| `aggregated_daily_revenue.csv` | Derived — daily revenue per restaurant, zero-filled (output of `aggregate.py`). |
| `features.csv` | Derived — full model feature matrix (output of `features.py`). |

### Critical data pitfalls

- **Double-counting** — joining `sale_customer` to `sale` and summing `amount` duplicates revenue. Sum revenue only from `sale.csv`.
- **Wrong date grain** — timestamps carry timezone offsets; truncate to date with `.dt.normalize()` (Python) or `substring(0, 10)` (JS).
- **Dropping zero days** — days with no sales are simply absent from raw data. `aggregate.py` fills them with R0. Skipping this badly distorts lag/rolling features.
- **Target grain** — the forecast target is `SUM(amount)` per restaurant per calendar day, not per bill or per customer.

## Forecasting pipeline

A gradient-boosted (XGBoost) daily-revenue model. Run the steps in order from the `forecast/` directory; each writes into `data/` or `forecast/output/`.

```bash
cd forecast

python aggregate.py    # 1. Raw sales → data/aggregated_daily_revenue.csv (zero-filled)
python features.py     # 2. Aggregated → data/features.csv (calendar, holiday, lag, rolling, ...)
python train.py        # 3. Tune, train, evaluate, and write forecast/output/july_forecast.csv
```

`explore.py` is a standalone scratch script for inspecting the raw sales file (date parsing, day-of-week revenue patterns); it is not part of the production pipeline.

### Model approach

- **Algorithm** — XGBoost regressor with `reg:absoluteerror` (MAE) loss on the raw ZAR target, for robustness to outlier trading days.
- **Features (45)** — calendar (day-of-week, payday markers, month position), SA holidays (public, school, bridge days, holiday depth, back-to-school), revenue lags (7/14/21/28/364/365/728 days), rolling means & std (7/14/28/90), same-week-last-year average, transaction/voucher signals, province encoding, per-restaurant scale (median revenue), and YoY growth. Full list in [forecast/output/model_metadata.json](forecast/output/model_metadata.json).
- **Sample weighting** — July and school-holiday rows are up-weighted; per-restaurant inverse-median weighting equalises attention across large and small outlets.
- **Hyperparameter search** — Optuna TPE, 100 trials, optimising MAPE across July-aligned time-series CV folds (Jul 2023, Jul 2024, Jun 2025).
- **Leakage guards** — rolling/lag features use `shift(1)`; the per-restaurant scale feature is computed from historical data only; forecast-row lags are filled via a cascade of longer lags then rolling proxies.
- **Validation** — held-out July 2024. Current metrics: **MAE ≈ R7,600**, **MAPE ≈ 29.6%** (near-zero days filtered). MAPE excludes days below 5% of a restaurant's median revenue to avoid noise blow-up.

### Forecast CSV validation checklist

`train.py` asserts all of these before writing:

- Exactly **682 rows** (excluding header).
- `restaurant_id` covers every integer **1–22**.
- `date` covers every day **2025-07-01 → 2025-07-31** (31 unique dates).
- `predicted_revenue` is numeric and **non-negative**.
- No duplicate `(restaurant_id, date)` pairs; no NaNs.

## Dashboard

A React 19 + Vite 8 app (Recharts for charts) targeting **non-technical restaurant managers** — no model diagnostics in the main view.

```bash
cd dashboard

npm install

# 1. Build the dashboard data bundle from the CSVs.
#    Reads data/aggregated_daily_revenue.csv, forecast/output/july_forecast.csv,
#    and data/sa_school_holidays.csv → public/sales_data.json.
#    Re-run whenever the data or forecast changes.
node scripts/preprocess.mjs

# 2. Start the dev server
npm run dev

# Production build
npm run build
```

### Dashboard features

- **Sales history** — daily revenue trend with enough context to see recent performance.
- **Forecast layer** — July 2025 forecast, clearly labelled, toggleable, and visually distinct from actuals.
- **Restaurant navigation** — switch between all 22 restaurants plus an "All" aggregate view.
- **Financial KPIs** — typical-day revenue, forecast month total, weekend vs weekday split, latest trading run-rate — all labelled in ZAR.
- **School-holiday markers** and a **restaurant ranking** panel.

# 0. Place all source CSVs in data/

# 1. Build the forecast
cd forecast
python aggregate.py
python features.py
python train.py            # writes forecast/output/july_forecast.csv

# 2. Build and run the dashboard
cd ../dashboard
npm install
node scripts/preprocess.mjs # writes public/sales_data.json
